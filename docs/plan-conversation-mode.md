# แผน: โหมดสนทนา JARVIS (ทักทาย + คุยต่อเนื่องหลายเทิร์น)

## Context (อยากได้อะไร)
ตอนนี้หน้าแรก (HUD) เป็น "ถามทีละคำถาม" — พูด/พิมพ์ 1 ครั้ง ได้คำตอบ 1 กล่อง คำถามใหม่ทับของเดิม
และถ้าพูดว่า "สวัสดี / hello jarvis" ระบบส่งเข้า RAG → ตอบ "ไม่พบเคสที่เกี่ยวข้อง" (ประสบการณ์แข็ง)

**เป้าหมาย:** คุยเป็นบทสนทนาแบบนี้ได้

```
user  : hello jarvis
jarvis: สวัสดีครับ มีอะไรให้รับใช้ครับ        ← ไม่เข้า RAG, ตอบทันที + พูดออกเสียง
user  : มอเตอร์ไหม้
jarvis: (คำตอบจากเคสจริง + citation)          ← เข้า RAG ตามเดิม
user  : แล้วใช้อะไหล่อะไรบ้าง
jarvis: (ตอบต่อเนื่อง รู้ว่า "อะไหล่" หมายถึงเคสมอเตอร์เมื่อกี้)   ← เฟสหลัง (ขั้น 4)
```

## ขอบเขต: 2 ไฟล์หลัก — `backend/app.py` (หรือ rag.py) + `frontend/src/components/Landing.jsx`

---

### ขั้น 1 — เลนทักทาย/คุยเล่น (backend)

จุดแทรก: ก่อนเรียก `rag.answer()` ใน `/api/ask` (หรือหัว `rag.answer` เอง — เสนอไว้ที่ app.py
เพราะเป็นเรื่อง "การสนทนา" ไม่ใช่ "การค้นความรู้")

- ตัวจับ greeting แบบ string match (0ms, deterministic — สไตล์เดียวกับ `_rel_min_for`):
  ```python
  _GREET_RE = re.compile(
      r"^(hello|hi|hey|เฮลโล่?|ฮัลโหล|สวัสดี|หวัดดี|ดีครับ|ดีค่ะ)[\s,!.]*"
      r"(jarvis|จาร์วิส|จาวิส)?[\s!.ครับค่ะ]*$", re.I)
  ```
  ต้องเป็น **ทั้งประโยค** (`^...$`) — "สวัสดีครับ มอเตอร์ไหม้แก้ยังไง" ต้องไม่เข้าเลนนี้
  และรองรับคำที่ Whisper ถอด "hello jarvis" เพี้ยนเป็นไทย (เฮลโล่ จาร์วิส / ฮัลโหล จาวิส)
- เจอ greeting → ตอบ canned ทันที (สุ่มจากลิสต์ 3-4 แบบให้ไม่จำเจ):
  `{"answer": "สวัสดีครับ มีอะไรให้รับใช้ครับ", "citations": [], "greeting": true, "mock": false}`
  **ไม่เรียก LLM เลย** — เร็ว (<10ms), ไม่มีความเสี่ยงมโน, ไม่เปลือง GPU
- เพิ่มลาก่อน/ขอบคุณด้วย: "ขอบคุณ/บาย/แค่นี้แหละ" → "ยินดีครับ เรียกผมได้ตลอดครับ"
- ไม่ log ลง `ask_history.jsonl` (กัน history รกด้วยคำทักทาย) — หรือ log ก็ได้ ต้องตัดสินใจ

### ขั้น 2 — UI บทสนทนา (Landing.jsx)

- เปลี่ยน state `answer` (กล่องเดียว) → `turns` (array ของ `{role: 'user'|'jarvis', text, citations, model, seconds}`)
- แสดงเป็น transcript ใต้วง HUD: bubble ผู้ใช้ชิดขวา / JARVIS ชิดซ้าย (โครงเดียวกับ
  `.hud-answer` เดิม แต่ append ไม่ทับ) — เลื่อน scroll ลงล่างอัตโนมัติเมื่อมีเทิร์นใหม่
- คำถามจากไมค์: ตอนถอดเสียงเสร็จ push เทิร์น user ทันที (เห็นว่าระบบได้ยินอะไร) แล้วค่อย push เทิร์น jarvis เมื่อตอบ
- ปุ่ม "ล้างบทสนทนา" เล็กๆ มุมบนของ transcript
- CSS ใหม่ ~30 บรรทัด (bubble 2 สี) — ที่เหลือ reuse ของเดิม

### ขั้น 3 — Hands-free ต่อเนื่อง (Landing.jsx)

ให้คุยต่อได้โดยไม่ต้องแตะไมค์ทุกรอบ:

- **auto-listen:** หลัง SPEAKING จบ (player `onended` / `u.onend`) → เปิดไมค์รอบใหม่อัตโนมัติ
  ถ้าอยู่ใน "โหมดสนทนา" (toggle เปิด/ปิดได้ — default เปิดเมื่อผู้ใช้เริ่มด้วยเสียง, ปิดเมื่อเริ่มด้วยพิมพ์)
- **auto-stop เมื่อเงียบ:** ใช้ analyser ที่มีอยู่แล้ววัด RMS — เงียบต่อเนื่อง ~2.5 วิ → `mr.stop()` เอง
  (ตอนนี้ต้องแตะหยุดเอง) + เพดานอัดสูงสุด ~15 วิ กันค้าง
- **ทางออกจากลูป:** เงียบ 2 รอบติด (ไม่ได้ยินอะไรเลย) หรือผู้ใช้พูดคำลา หรือแตะปุ่มหยุด → กลับ idle
- ความเสี่ยง: ไมค์เปิดค้าง + TTS ลำโพงดังเข้าไมค์เอง (echo) → เริ่มอัดหลังเสียงตอบจบเท่านั้น (ไม่ฟังแทรก)

### ขั้น 3.5 — Wake word "hello jarvis" (Landing.jsx, Chrome/Edge เท่านั้น)

เป้าหมายจริงของผู้ใช้: พูด "hello jarvis" ตอนไมค์ยังไม่เปิด → ระบบตื่นมาทัก → ถามต่อได้เลย
(ไม่ต้องแตะปุ่มก่อน)

- ใช้ `webkitSpeechRecognition` (มีในตัว Chrome/Edge ฟรี) เป็น **ตัวดักคำปลุกอย่างเดียว**
  ตั้ง `continuous = true`, `interimResults = true`, `lang = 'th-TH'` (จับได้ทั้งไทย+อังกฤษพอควร)
- ฟัง transcript แบบ interim → ถ้ามี "hello jarvis / เฮลโล่ จาร์วิส / สวัสดีจาร์วิส" (regex หลวมๆ
  ตัวเดียวกับ `_GREET_RE` แต่ฝั่ง JS) → หยุด SR → ทัก "สวัสดีครับ" (canned + พูด) → เปิดไมค์ Whisper ตัวจริง (`toggleMic`)
- **คำถามจริงยังผ่าน Whisper เดิม** — browser SR ใช้แค่จับคำปลุก (ไม่เอาความแม่นมาตอบเคส)
- state ใหม่: `wakeMode` (toggle "ปลุกด้วยเสียง 🎙️" บนจอ) — เปิด = SR ฟังคำปลุกตลอด, ปิด = พฤติกรรมเดิม
- จัดการ SR ที่ Chrome auto-stop เองทุก ~60 วิ → `onend` → ถ้ายัง wakeMode อยู่ให้ `.start()` ใหม่ (กันตายเงียบ)
- echo: ตอน JARVIS พูดตอบ ต้อง **หยุด SR ชั่วคราว** ไม่งั้นจับเสียงตัวเองเป็นคำปลุกวนลูป

**ข้อจำกัดที่ต้องรู้ (บันทึกไว้):**
- ทำงานแค่ **Chrome/Edge** — Firefox/Safari ไม่มี `webkitSpeechRecognition` → ต้อง feature-detect
  แล้วซ่อน toggle นี้ (fallback = auto-listen ขั้น 3 ที่ทำงานทุกเบราว์เซอร์)
- Chrome ส่งเสียงไป Google ประมวลผล (cloud) — privacy: ยอมรับได้สำหรับ demo, แต่ไม่ใช่ offline
- ไม่ใช่ wake-word engine จริงจัง (เทียบ Porcupine/Picovoice) — แม่นพอสำหรับคำปลุกสั้นๆ ในที่เงียบ
  ที่เสียงดัง/หลายคนพูดจะ false trigger บ้าง

### ขั้น 3.6 — กัน noise ยิง RAG รัว (สำคัญ — state machine 2 โหมดฟัง)

ปัญหา: ถ้าตอบจบแล้วฟังค้าง noise ในห้องจะถูกถอด → RAG → "ไม่พบเคส" วนไม่จบ

**หลักการ: แยก 2 โหมดฟังชัดเจน**
- **SLEEPING (ฟังคำปลุกอย่างเดียว):** browser SR ฟังตลอด แต่ reactเฉพาะ "hello jarvis"
  — noise ไม่ใช่คำปลุก = เมิน ตรงนี้ noise ทำอะไรไม่ได้โดยธรรมชาติ (default state ระหว่างเทิร์น)
- **LISTENING (Whisper active):** เปิดเฉพาะหลังทัก/หลังตอบ — มีตัวปิด 3 ชั้น:
  1. **auto-stop เงียบ ~2 วิ** — RMS จาก analyser เดิม ต่ำกว่าเกณฑ์ต่อเนื่อง → `mr.stop()` (+ เพดาน 15 วิ)
  2. **กรองก่อนส่ง:** transcript ว่าง/สั้นกว่า ~2-3 ตัวอักษร (Whisper ถอด noise/เงียบได้แบบนี้)
     → **ทิ้งเงียบ ไม่เรียก RAG ไม่พูด "ไม่พบเคส"** → กลับ SLEEPING
  3. **หน้าต่างถามต่อจำกัด:** ตอบจบเปิดฟังต่อ ~5 วิ; ไม่ได้ยินคำพูดจริง → กลับ SLEEPING
     (ไม่วนฟังไม่จบ — ต้อง "hello jarvis" ปลุกใหม่)

**state machine:**
```
SLEEPING ──"hello jarvis"──> GREET ──> LISTENING(Whisper)
    ▲                                        │
    │                              พูดจริง(ผ่านตัวกรอง)? ─ใช่─> THINKING ─> SPEAKING ─> LISTENING(5s)
    └──────── noise / เงียบ / ไม่มีคำปลุก ─────┘                                    (เงียบ→SLEEPING)
```

- ตอน SPEAKING (JARVIS พูด) → หยุดทั้ง SR และ Whisper กัน echo จับเสียงตัวเอง
- ถ้า RAG ตอบ "ไม่พบเคส" จาก**คำถามจริง** (ไม่ใช่ noise) → แสดง/พูดตามปกติ (นั่นคือคำตอบที่ถูก)
  ต่างจาก noise ที่ถูกกรองทิ้งตั้งแต่ชั้น 2 ก่อนถึง RAG

### ขั้น 4 (เฟสหลัง ทำทีหลังได้) — จำบริบทหลายเทิร์น

- frontend ส่ง `history: [{role, text}, ...]` (3-4 เทิร์นล่าสุด) ไปกับ `/api/ask`
- backend: คำถามสั้นที่ไม่มีศัพท์ช่าง (เช่น "แล้วใช้อะไหล่อะไร") + มี history → ให้ LLM ย่อ
  คำถาม+บริบทเป็น query เดียวก่อนค้น (query rewrite 1 ครั้ง) แล้วเข้า RAG ตามปกติ
- แยกเป็นแผน/คอมมิตของตัวเอง — อย่ามัดรวมกับขั้น 1-3 (ขั้น 1-3 ใช้งานได้ด้วยตัวเองแล้ว)

## ความเข้ากันได้
- `/api/ask` โครง response เดิมครบ (เพิ่ม field `greeting` เฉยๆ) → หน้า `/ask` (AskDemo) ไม่พัง
  และได้เลนทักทายฟรีไปด้วย
- ประวัติถาม-ตอบ + sidebar เดิมทำงานเหมือนเดิม (greeting ไม่ log ตามข้อตัดสินใจ)
- โหมดพิมพ์ถาม ยังใช้ได้ตลอด — transcript เดียวกัน
- mock lane (`_MOCK_*`) ไม่แตะ

## Verification
1. พูด/พิมพ์ "hello jarvis", "สวัสดีครับ", "เฮลโล่ จาร์วิส" → ตอบทักทายทันที ไม่มี citation, ไม่แตะ RAG (ดู log ไม่มี `[RAG]`)
2. "สวัสดีครับ มอเตอร์ไหม้แก้ยังไง" (ทัก+ถามในประโยคเดียว) → ต้องเข้า RAG ไม่ใช่เลนทักทาย
3. คุย 3 เทิร์น: ทักทาย → "มอเตอร์ไหม้" → "สายพาน Ejector" — transcript แสดงครบทุกเทิร์น ไม่ทับกัน citations ตรงเคส
4. โหมดเสียง: ทักด้วยเสียง → ตอบเสร็จไมค์เปิดเอง → ถามต่อโดยไม่แตะ → เงียบ 2 รอบ → กลับ idle เอง
5. regression เดิม: hit/miss/agg ผ่านครบ (เลนทักทายต้องไม่กินคำถามช่าง — "สวัสดี" อยู่ใน `REL_MIN_OTHER` เดิมด้วยซ้ำ ตอนนี้แค่ตอบสวยขึ้น)

## ตัดสินใจแล้ว (จากผู้ใช้)
1. greeting **ไม่ log** ลงประวัติ ✓
2. auto-listen = **มี toggle ชัดๆ** บนจอ (ชิป "โหมดสนทนา") ✓
3. wake word "hello jarvis" = **เอา** → ทำผ่าน `webkitSpeechRecognition` (ขั้น 3.5),
   รับข้อจำกัด Chrome/Edge เท่านั้น + feature-detect ซ่อน toggle บนเบราว์เซอร์ที่ไม่รองรับ

## ไฟล์ที่แก้
- `backend/app.py` — เลนทักทาย/ลา ก่อนเรียก RAG (~25 บรรทัด)
- `frontend/src/components/Landing.jsx` — transcript หลายเทิร์น + auto-listen/auto-stop
  + wake word ผ่าน webkitSpeechRecognition (~180 บรรทัด)
- `frontend/src/index.css` — bubble บทสนทนา + toggle โหมดสนทนา/ปลุกด้วยเสียง (~40 บรรทัด)

## ลำดับทำ (เห็นผลไว → ซับซ้อน)
1. ขั้น 1 (backend ทักทาย) + ขั้น 2 (transcript) — เห็นผลไวสุด ไม่แตะไมค์เดิม
2. ขั้น 3 (auto-listen ต่อเนื่อง)
3. ขั้น 3.5 (wake word "hello jarvis") — pièce de résistance ที่ผู้ใช้อยากได้
4. ขั้น 4 (จำบริบท) — เฟสหลัง แยกคอมมิต
